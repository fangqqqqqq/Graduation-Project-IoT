# 城市智能巡检小车系统设计报告（UI与数据库篇）

## 1. 上位机 UI 界面功能模块说明
本系统的上位机软件基于 Python 的 Tkinter 库开发，采用深色科技风主题（Dark Theme），以保障巡检人员在长时间监控下的视觉舒适度。整体界面划分为“左侧视频监控区”与“右侧核心控制面板”两大模块。

### 1.1 左侧视频监控区 (Video Display Area)
* **实时画面画布 (Canvas)**：以 640x480 的分辨率展示 ESP32-CAM 传回的实时视频流。当开启 AI 识别时，画面上会实时叠加目标检测框（Bounding Box）、类别标签及置信度。
* **状态指示栏**：位于视频顶部，显示“● 信号正常 (LIVE)”或“❌ 信号中断”等系统状态。
* **[🔄 重启视频流] 按钮**：用于在网络波动导致视频卡顿时，手动重置 OpenCV 的 VideoCapture 线程。

### 1.2 右侧核心控制面板 (Control Panel)
控制面板采用紧凑型网格布局，由上至下划分为 5 个子功能区：

1. **[📡 连接配置] 区**：
   * **端口输入框**：填入 STM32 连接的串口号（如 COM12）。
   * **[🔗 连接 / 断开] 按钮**：控制串口通信的开启与关闭。
   * **连接状态标签**：实时显示当前连接状态。

2. **[📊 环境监测] 区**：
   * 实时解析并显示小车底层传感器传回的三项核心数据：**距离 (📏 cm)**、**温度 (🌡️ ℃)**、**湿度 (💧 %)**。

3. **[🎮 核心控制] 区**：
   * **小车底盘运动按键**：采用经典的 `Q-W-E` 与 `A-S-D` 布局。包含 **[↺ 左旋]、[⬆ 前进]、[↻ 右旋]、[⬅ 左转]、[🛑 急停]、[➡ 右转]、[⬇ 后退]**。按键均绑定了“按下发送指令、松开发送停止指令”的状态机逻辑，保证操控精准度。
   * **硬件模式按键**：包含 **[🛡️ 自动避障模式]** 和 **[🛤️ 黑线循迹模式]**，用于一键切换 STM32 底层的自主运动逻辑。
   * **云台控制按键**：包含 **[☁ 云台上升]** 和 **[☁ 云台下降]**，通过向舵机发送指令调整摄像头的俯仰角。

4. **[🤖 AI 视觉] 区（上位机智能功能）**：
   * **[👁️ 目标识别: 开/关]**：一键启停 YOLOv8 推理线程。
   * **[🚀 自动巡逻 (Python)]**：上位机接管小车控制权，当检测到坑洼等障碍时，自动发送避障逻辑指令。
   * **[💡 补光] 滑块**：0-255 无极调节 ESP32-CAM 自带闪光灯的亮度。
   * **[🔄 画面旋转] 按钮**：支持 0°/90°/180°/270° 旋转，适应不同安装角度的摄像头。

5. **[数据中心] 区**：
   * **[📊 历史数据中心] 按钮**：点击弹出历史数据查阅子窗口，支持通过 Treeview 表格查看环境日志和 AI 警报记录。

## 2. 数据库结构分析与优化方案

本项目采用轻量级的 SQLite 数据库 (`patrol_record.db`) 进行数据持久化，以满足边缘计算场景下的低开销需求。

### 2.1 当前数据库结构 (V1.0)
目前数据库包含两张独立的数据表：
1. **`sensor_logs` (环境数据表)**
   * 字段：`id`, `timestamp` (时间戳), `distance` (距离), `temperature` (温度), `humidity` (湿度), `note` (触发记录的备注，如“温度突变”、“定时记录”)。
2. **`detection_logs` (巡检事件表)**
   * 字段：`id`, `timestamp` (时间戳), `object_type` (目标类别，如 can/pothole), `confidence` (置信度), `image_path` (现场快照的本地存储路径)。

### 2.2 当前结构存在的缺陷分析
1. **数据孤岛效应**：环境数据与 AI 识别事件相互独立。当 AI 记录到一个坑洼时，无法直接关联当时的温湿度环境，不利于后续的“环境与路面病害关联性分析”。
2. **缺乏宏观任务管理**：目前的数据是按时间线平铺的“流水账”，无法区分数据属于哪一天的哪一次巡检任务。
3. **缺乏业务闭环**：发现的异常事件仅被记录，没有跟进状态（如待处理、已修复），无法满足真实的城市巡检派单需求。

### 2.3 下一步优化方案规划 (V2.0)
在后续的项目迭代中，计划对数据库进行以下重构：
1. **引入任务会话表 (`patrol_sessions`)**：每次点击“开启自动巡逻”时生成一个唯一的 `session_id`，将沿途的环境日志和警报记录全部挂载到该 ID 下。
2. **事件表数据融合 (Data Snapshot)**：在 `detection_logs` 中增加 `snapshot_temp` 和 `snapshot_humi` 字段。在触发 YOLO 警报存图的瞬间，同步捕获当时的温湿度写入该表。
3. **引入状态机**：为事件表增加 `status` 字段（默认：待处理）。在历史数据中心 UI 中增加“右键标记为已修复”功能，形成完整的物联网业务闭环。
